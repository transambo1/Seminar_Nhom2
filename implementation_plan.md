# StormShield MVP - System Architecture Plan

This document outlines a clean, scalable, yet realistic microservice architecture for the **StormShield** seminar project. The goal is to provide a comprehensive distributed system that remains achievable within the timeframe of a specialized course.

## 1. Final List of Microservices

To avoid overengineering while strictly meeting the requirements, the system will consist of **six** core microservices (including infrastructure):

1. **`discovery-service`** (Infrastructure)
2. **`api-gateway`** (Infrastructure)
3. **`user-service`** (Business)
4. **`map-service`** (Business)
5. **`incident-service`** (Business)
6. **`alert-service`** (Business)

> [!TIP]
> **Why combine Reports and Requests?**
> Instead of separate "Report" and "Rescue" services, they are unified under an **`incident-service`**. They share similar workflows and data structures (spatial coordinates, user IDs, media, and state machines like PENDING -> REVIEWED/RESOLVED). This drops the service tally, making the project easier to host and deploy locally for a seminar.

---

## 2. Responsibility of Each Microservice

### 🛡️ Infrastructure Services
- **`discovery-service` (Eureka)**: Acts as the service registry. All other services will register themselves here so they can be discovered by name instead of hardcoded IP addresses.
- **`api-gateway` (Spring Cloud Gateway)**: The single entry point for the frontend. Handles CORS, route forwarding, rate limiting, and validates JWTs before passing requests to downstream services.

### 💼 Business Services
- **`user-service`**: Handles Authentication (Login/Register), issues JWT access tokens, and manages User Roles (`CITIZEN`, `RESCUER`, `ADMIN`, `AUTHORITY`).
- **`map-service`**: Manages the static spatial layers. Responsible for CRUD operations on Shelters, computing nearby shelters, and exposing pre-defined "Safe Road" or "Flood Zone" GeoJSON layers.
- **`incident-service`**:
  - **Hazard Reports**: Receives reports from users (text, photo, optional GPS). Admins review, and approved items are served back to clients as hazard pins.
  - **Rescue Requests**: Receives SOS calls from users. Tracks status (`PENDING`, `ASSIGNED`, `IN_PROGRESS`, `RESOLVED`, `CANCELLED`).
- **`alert-service`**: Responsible for storing emergency alerts generated by authorities and persisting notification histories. Broadcasts live alerts to users connected via WebSocket.

---

## 3. Mandatory vs. Optional Services (for MVP Demo)

**Mandatory:** 
- `discovery-service`
- `api-gateway`
- `user-service`
- `map-service`
- `incident-service`

**Highly Recommended (Adds the "WOW" Factor):**
- `alert-service` (Real-time WebSockets look amazing in demos. If time is severely limited, notifications could technically be polled straight to `incident-service`, but an event-driven `alert-service` is much more impressive).

---

## 4. Recommended Implementation Order

To ensure a smooth integration, develop linearly layer by layer:

1. **Phase 1 (Foundation):** Setup `discovery-service` and `api-gateway`. Verify routing works with dummy endpoints.
2. **Phase 2 (Security):** Build `user-service`. Implement login, JWT creation, and write a filter in the Gateway to validate the token.
3. **Phase 3 (Core View):** Build `map-service`. Setup the React/Leaflet frontend. Get the map rendering with dummy shelter pins and static flood zone polygons.
4. **Phase 4 (Interactions):** Build `incident-service`. Implement the frontend forms to submit / track Rescue Requests and Hazard Reports.
5. **Phase 5 (Event-Driven Polish):** Build `alert-service` + RabbitMQ. Send a push notification when a status changes.

---

## 5. Database Selection

*Using Polyglot Persistence demonstrates strong architectural decision-making:*

- **`user-service`**: **MySQL** or **PostgreSQL**. Relational data is perfect for strict schemas like User credentials, Roles, and flat profiles.
- **`map-service`**: **PostgreSQL with PostGIS**. Crucial for spatial queries (e.g., `ST_Distance` to find shelves within X radius).
- **`incident-service`**: **MongoDB**. Schema-less design makes it easy to support reports with optional varying metadata, images, and changing state-tracking documents.
- **`alert-service`**: **MongoDB**. Very fast write speeds, ideal for inserting high volumes of notification history records.

---

## 6. Synchronous Inter-Service Calls (OpenFeign/REST)

Use **Synchronous** calls when the user cannot proceed without the result:
- **`api-gateway` -> `[Any Service]`**: Standard HTTP routing.
- **`incident-service` -> `user-service`**: When viewing an incident, the incident service might need to synchronously fetch the user's name/phone to display to the Rescue Team.
- **`incident-service` -> `map-service`**: (Optional) To validate if a submitted incident falls inside an existing known geographic boundary.

---

## 7. Asynchronous Inter-Service Calls (RabbitMQ)

Use **Asynchronous** (Message Broker) events for decoupling side-effects:
- **Hazard Approved Event**: When an admin in `incident-service` approves a community report, it publishes an event. `alert-service` consumes it -> pushes WebSocket warnings to users looking at that map area.
- **Status Changed Event**: When a rescue request advances to `ASSIGNED`, `incident-service` publishes an event. `alert-service` consumes it -> saves a notification to the database -> pushes the update live to the victim's phone.

---

## 8. Features to Mock in MVP

To keep the scope realistic and avoid API/infrastructure bloat:
1. **Real-time GPS Tracking**: Rather than a live WebSocket coordinate stream for rescuers, mock it by letting a rescuer click a "Check In" button to manually update their location every few minutes.
2. **External Weather Maps**: Don't use a complicated paid meteorology API. Mock "flood zones" with static `GeoJSON` files created at geojson.io, loaded by your `map-service`.
3. **SMS / Mobile Push**: Skip Twilio / Firebase FCM. Use a simple "Notification Bell" and an internal WebSocket topic on the web app to prove the concept without paid integrations.
4. **Image Uploads (AWS S3)**: Avoid Cloud setup time; for MVP, store images as Base64 strings directly in MongoDB, or save them to an `/uploads` folder mounted via Docker.
5. **Routing/Navigation Algorithm**: Don't write Dijkstra map routing. Just return predefined simple LineStrings to represent "Safe Roads" for the demo.
